# `hr_employees` — Employee master

Phase 1A. Employee master record plus the three reference tables it depends on
(companies, departments, positions). PII fields (`national_id`, `salary`) are
encrypted at rest with `pgcrypto` and masked for callers that don't hold
`hr_employees.reveal_pii`.

## Tables (owned)

| Table | Purpose |
|---|---|
| `companies` | Legal entities. Read-only dropdown here; full CRUD belongs to `settings` (Phase 1B). |
| `departments` | Per-company org units. Read-only dropdown here. |
| `positions` | Job titles + level (1–10). Read-only dropdown here. |
| `employee_code_sequences` | Per-(company, year) counter backing `generate_employee_code()`. |
| `employees` | One row per person. `national_id` + `salary` are `BYTEA` after migration 0003 (pgp_sym_encrypt / pgp_sym_decrypt). |

Schema: [`backend/migrations/0002_hr_master.up.sql`](../../../migrations/0002_hr_master.up.sql).
Encryption conversion: [`backend/migrations/0003_hr_encrypt.up.sql`](../../../migrations/0003_hr_encrypt.up.sql).

## Employee code format

Generated by the SQL function `generate_employee_code(company_id)` at INSERT
time:

```
<company_code>-<YY><NNNN>    e.g.  TMT-260001
```

The per-(company, year) counter lives in `employee_code_sequences` and is
bumped atomically via `INSERT … ON CONFLICT UPDATE`.

## HTTP endpoints

Mounted under `/api/v1/hr` in `cmd/api/main.go`. All require an authenticated
session; specific permissions per endpoint:

| Method | Path | Permission |
|---|---|---|
| `GET` | `/hr/companies` | session only |
| `GET` | `/hr/departments?company_id=…` | session only |
| `GET` | `/hr/positions` | session only |
| `GET` | `/hr/employees` | `hr_employees.read` |
| `GET` | `/hr/employees/:id` | `hr_employees.read` |
| `POST` | `/hr/employees` | `hr_employees.write` |
| `PATCH` | `/hr/employees/:id` | `hr_employees.write` |
| `POST` | `/hr/employees/:id/terminate` | `hr_employees.terminate` |

## Permissions

Seeded by the RBAC seeder:

- `hr_employees.read` — list + detail
- `hr_employees.write` — create + update
- `hr_employees.terminate` — terminate an employee
- `hr_employees.reveal_pii` — return `national_id` + `salary` in plaintext.
  Without this permission the handler masks both fields before responding,
  regardless of how they're stored.

## PII masking

`applyPIIMask` in `employees.go` runs on every response path:

- `national_id` — all-but-last-digit replaced with `•`
- `salary` — replaced with `•••••`

The decrypt only happens server-side; masked values never leave the process.
When a caller with `reveal_pii` requests the unmasked form, the handler emits
an audit entry of action `hr_employees.reveal_pii` with the entity id — so
every reveal is retained in the chain.

## Encryption key

`PII_ENCRYPTION_KEY` is required in production. In `development` a safe
fallback is used (see `internal/config/config.go`). Rotation is not yet
automated; when it lands it will re-encrypt rows in a single transaction
using both keys, then drop the old one.

## Audit

Every mutation calls `audit.Write` with `before` and `after` JSON. The
forward-chained SHA-256 hash in `audit_log.row_hash` makes tampering detectable.
